# GRACE Skill: Семантическая разметка кода

## Назначение

Этот скилл обеспечивает создание и поддержку семантической разметки кода, делая его понятным для AI-агентов через ANCHOR и контракты.

---

## Ключевые принципы

1. **Контракт-первый** — код должен соответствовать контракту, не наоборот
2. **Явная семантика** — каждый блок имеет адрес (ANCHOR) и смысл (контракт)
3. **Наблюдаемость** — логи позволяют восстановить траекторию выполнения
4. **Запрет разрушения** — `@ForbiddenChanges` защищает критичную логику
5. **Без догадок** — при неопределённости — сохранить поведение, пометить `@SEMANTIC_AMBIGUITY`

---

## Обязательные элементы контракта

### 1. ANCHOR (Якорь)

Уникальный идентификатор блока в формате `UPPER_SNAKE_CASE`.

**Правило размещения:**
- `// [START_ANCHOR_ID]` — маркер начала блока
- `ANCHOR: ANCHOR_ID` — внутри контракта
- `// [END_ANCHOR_ID]` — маркер конца блока
- `anchor` в `logLine()` — для привязки логов

### 2. PURPOSE (Назначение)

Одна фраза — зачем существует этот блок.

### 3. @PreConditions (Предусловия)

Что должно быть истинно ДО выполнения.

### 4. @PostConditions (Постусловия)

Что гарантируется ПОСЛЕ выполнения.

### 5. @Invariants (Инварианты)

Что не должно нарушаться НИКОГДА.

### 6. @SideEffects (Побочные эффекты)

Побочные эффекты (или "нет побочных эффектов").

### 7. @ForbiddenChanges (Запреты)

Что запрещено менять семантически.

### 8. @AllowedRefactorZone (Опционально)

Что можно безопасно переписывать.

## Шаблон контракта

### TypeScript / JavaScript

```typescript
// [START_ANCHOR_ID]
/*
 * ANCHOR: ANCHOR_ID
 * PURPOSE: Одна фраза — зачем существует эта функция.
 * 
 * @PreConditions:
 * - условие 1
 * - условие 2 (или "нет нетривиальных предусловий")
 * 
 * @PostConditions:
 * - гарантия 1 при успехе
 * - гарантия 2 при ошибке (если применимо)
 * 
 * @Invariants:
 * - инвариант 1 (то, что истинно до, во время и после)
 * - инвариант 2
 * 
 * @SideEffects:
 * - побочный эффект 1 (или "нет побочных эффектов")
 * 
 * @ForbiddenChanges:
 * - что запрещено менять без явного согласования
 * 
 * @AllowedRefactorZone:
 * - что можно безопасно переписывать
 */
async function example(): Promise<void> {
  logLine("module", "DEBUG", "functionName", "ANCHOR_ID", "ENTRY", { args: "..." });
  // реализация
  logLine("module", "DEBUG", "functionName", "ANCHOR_ID", "EXIT", { result: "..." });
}
// [END_ANCHOR_ID]
```

## Иерархия приоритетов

При конфликте приоритет следующий (от высшего к низшему):

1. **Контракт** — зафиксированный смысл блока
2. **Инварианты** — то, что не должно нарушаться никогда
3. **Постусловия** — гарантии результата
4. **Текущая реализация** — код, который можно переписывать

**Если реализация расходится с контрактом, агент обязан приводить реализацию в соответствие контракту, а не наоборот.**

---

## Запреты на разрушительный рефакторинг

### Категорически запрещено

- **Удалять "странную" логику без проверки её смысловой роли** — за ней может стоять неизвестное бизнес-ограничение
- **Менять поведение под видом cleanup** — рефакторинг не меняет семантику
- **Заменять бизнес-ограничения на "более стандартные" паттерны** — неочевидные ограничения могут быть критичны
- **Выдумывать недостающий смысл** — если он не закреплён явно, его нет
- **Менять контракт под реализацию** — только наоборот

### Требование к рефакторингу

При любом рефакторинге агент обязан:
1. Проверить, не нарушены ли инварианты
2. Проверить, обеспечены ли постусловия
3. Проверить, не появились ли новые побочные эффекты
4. Проверить, не нарушены ли запреты из `@ForbiddenChanges`
5. Убедиться, что поведение соответствует контракту

## Работа при неполном контексте

### Правило отсутствия догадок

**Если контракт, якорь или смысл блока неочевидны, агент НЕ ДОЛЖЕН молча додумывать поведение как ему кажется "логичным".**

### Обязательные действия при неопределённости

Агент обязан:
1. **Сохранить текущую семантику максимально бережно**
2. **Минимизировать изменения**
3. **Явно пометить место как требующее уточнения:**
   ```python
   # @SEMANTIC_AMBIGUITY: Требуется уточнение смысла блока
   # @PRESERVED_BEHAVIOR: Поведение сохранено как есть без изменений
   ```
4. **Не разрушать существующие ограничения**

## Работа с legacy-кодом

### Принцип

**Если код плохой, устаревший или запутанный — это НЕ повод ломать его смысл.**

### Процесс работы с legacy

1. **Извлечь или реконструировать контракт текущего поведения**
2. Зафиксировать контракт явно (даже если поведение кажется "неправильным")
3. Только после этого предлагать переписывание реализации
4. Любое изменение смысла требует отдельного согласования

### Шаблон для legacy-кода без контракта

```python
# [START_LEGACY_BLOCK_ID]
# @LEGACY_CONTRACT_RECONSTRUCTED:
# PURPOSE: (описать наблюдаемое поведение)
# @PreConditions: (описать выявленные предусловия)
# @PostConditions: (описать выявленные постусловия)
# @Ambiguity: (что осталось неясным)
# @OriginalBehavior: (ссылка на оригинальный код или коммит)
# ...
# [END_LEGACY_BLOCK_ID]
```

## Чеклист перед сдачей кода

### Обязательные проверки

- [ ] У каждой новой/изменённой функции есть `[START_…]` … `[END_…]`
- [ ] Контракт содержит ВСЕ обязательные поля
- [ ] ANCHOR_ID в START, ANCHOR:, END и logLine — одинаковый
- [ ] Инварианты не нарушены
- [ ] Постусловия обеспечены
- [ ] Запреты из `@ForbiddenChanges` соблюдены
- [ ] Новые побочные эффекты отсутствуют (или отражены в контракте)
- [ ] Legacy-код обработан по правилу реконструкции контракта
- [ ] Неопределённость помечена `@SEMANTIC_AMBIGUITY` вместо догадок

### Критерий готовности

Код считается готовым к ревью только при выполнении всех пунктов чеклиста.
