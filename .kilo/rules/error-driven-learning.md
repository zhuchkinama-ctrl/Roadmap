# Обучение на ошибках (Error-Driven Learning)

## Контекст проекта TrackHub

Этот документ описывает процесс создания и управления правилами для предотвращения повторных ошибок в проекте TrackHub (Java 17 + Spring Boot + Angular 17).

**Технологический стек:**
- **Backend**: Java 17, Spring Boot 3.3.x, Spring Data JPA, PostgreSQL 16, Spring Security + JWT
- **Frontend**: Angular 17+, TypeScript 5.x, PrimeNG, RxJS/Signals
- **Архитектура**: REST API, сервисный слой, репозитории, DTO

---

## Ключевой принцип

**Любая ошибка, замечание или неоднозначность — это сигнал для создания уточняющего правила. Система должна самообучаться, а не повторять одни и те же ошибки.**

---

## 1. Триггеры для создания правила

Правила создаются при:

| Триггер | Пример |
|---------|--------|
| **Ошибка в коде** | Нарушение контракта, баг, падение теста |
| **Замечание пользователя** | "Не то сделал", "Нужно по-другому" |
| **Нарушение стиля** | Не тот файл, не та структура |
| **Неоднозначность** | Агент ошибся в интерпретации |
| **Повторяющаяся проблема** | Одна ошибка >2 раз |
| **Code review замечание** | Отклонение на ревью |

## 2. Процесс создания правила

### 2.1 Алгоритм

```
ОШИБКА/ЗАМЕЧАНИЕ
       ↓
[Фиксация проблемы]
       ↓
[Анализ причины: почему произошло?]
       ↓
[Формулировка правила: что делать правильно?]
       ↓
[Классификация: к какой категории относится?]
       ↓
[Добавление в rules/*.md]
       ↓
[Обновление README.md]
       ↓
[Проверка: предотвращает ли правило повтор?]
```

### 2.2 Формат правила

```markdown
### RULE_[ID]: [Название]

**Проблема**: Что произошло (пример ошибки).

**Причина**: Почему агент ошибся.

**Решение**: Как нужно делать правильно.

**Пример**:
❌ Неправильно: ...
✅ Правильно: ...

**Применимость**: Когда применять это правило.

**Связанные RULE**: Другие правила по теме.
```

### 5.1 При обнаружении ошибки

Агент ОБЯЗАН:

1. **Зафиксировать ошибку** — что именно не так
2. **Проанализировать причину** — почему принято неверное решение
3. **Сформулировать правило** — как избежать в будущем
4. **Добавить правило** в соответствующий файл
5. **Обновить README.md** — добавить файл/правило в список

## 6. Примеры правил

### Пример 1: Нарушение контракта

```markdown
### RULE_CONTRACT_001: Проверять Invariants перед изменениями

**Проблема**: [2024-01-15] Агент добавил новый побочный эффект, не указав его в контракте.

**Причина**: Агент не проверил @SideEffects перед изменением кода.

**Решение**:
1. Перед любым изменением процедуры/функции — прочитать контракт
2. Проверить @SideEffects и @Invariants
3. Если новое поведение не подходит — обновить контракт ИЛИ отказаться от изменения

**Пример**:
❌ Неправильно:
```java
// [START_PROCESS_TRACK]
/*
 * ANCHOR: PROCESS_TRACK
 * PURPOSE: Обработка трека пользователя
 * @SideEffects: обновление трека в БД
 */
public TrackResult processTrack(String trackId, String userId) {
    // ... логика
    sendNotification(userId, "track_updated");  // новый побочный эффект без контракта
    return result;
}
// [END_PROCESS_TRACK]
```

✅ Правильно:
```java
// [START_PROCESS_TRACK]
/*
 * ANCHOR: PROCESS_TRACK
 * PURPOSE: Обработка трека пользователя
 * @SideEffects: обновление трека в БД, отправка уведомления пользователю
 */
public TrackResult processTrack(String trackId, String userId) {
    // ... логика
    sendNotification(userId, "track_updated");
    return result;
}
// [END_PROCESS_TRACK]
```

**Применимость**: При любом изменении существующей функции с контрактом.
```

### Пример 2: Неправильный файл

```markdown
### RULE_ARCH_001: Размещение бизнес-логики в сервисах

**Проблема**: [2024-01-16] Агент создал бизнес-метод в контроллере вместо сервиса.

**Причина**: Агент не проверил структуру проекта в project-context.md.

**Решение**:
1. Перед созданием нового метода — проверить структуру в project-context.md
2. Бизнес-логика -> src/main/java/org/homework/service/[Domain]Service.java
3. Контроллеры -> src/main/java/org/homework/controller/[Domain]Controller.java (только REST-эндпоинты)
4. DTO -> src/main/java/org/homework/dto/

**Пример**:
❌ Неправильно:
- src/main/java/org/homework/controller/TrackController.java: public Track calculateDiscount(String trackId)

✅ Правильно:
- src/main/java/org/homework/service/TrackService.java: public Track calculateDiscount(String trackId)
```

### Пример 3: Отсутствие валидации в контроллере

```markdown
### RULE_CONTROLLER_001: Использовать Bean Validation в контроллерах

**Проблема**: [2024-01-17] Агент не добавил @Valid для DTO в контроллере, что привело к пропуску невалидных данных.

**Причина**: Агент забыл про Bean Validation аннотации Spring.

**Решение**:
1. Всегда добавлять @Valid к параметрам DTO в контроллерах
2. Использовать аннотации валидации в DTO (@NotBlank, @Size, @Pattern)
3. Обрабатывать ValidationException в GlobalExceptionHandler

**Пример**:
❌ Неправильно:
```java
@PostMapping("/register")
public ResponseEntity<AuthResponse> register(@RequestBody RegisterRequest request) {
    // Нет валидации
    return ResponseEntity.ok(authService.register(request));
}
```

✅ Правильно:
```java
@PostMapping("/register")
public ResponseEntity<AuthResponse> register(@Valid @RequestBody RegisterRequest request) {
    return ResponseEntity.ok(authService.register(request));
}
```

**Применимость**: При создании или изменении REST-эндпоинтов в контроллерах.
```

### Пример 4: Неправильное использование @PreAuthorize

```markdown
### RULE_SECURITY_001: Проверять права доступа через PermissionService

**Проблема**: [2024-01-18] Агент использовал только @PreAuthorize без проверки прав на конкретные сущности.

**Причина**: Агент не учёл, что права доступа зависят от ролей OWNER/EDIT/VIEW.

**Решение**:
1. Для проверки ролей использовать @PreAuthorize("hasRole('ADMIN')")
2. Для проверки прав на сущности использовать PermissionService
3. Проверять права в сервисном слое, а не в контроллере

**Пример**:
❌ Неправильно:
```java
@PreAuthorize("isAuthenticated()")
@PutMapping("/tracks/{id}")
public ResponseEntity<TrackDto> updateTrack(@PathVariable Long id, @RequestBody TrackDto dto) {
    // Нет проверки прав на конкретный трек
    return ResponseEntity.ok(trackService.updateTrack(id, dto));
}
```

✅ Правильно:
```java
@PreAuthorize("isAuthenticated()")
@PutMapping("/tracks/{id}")
public ResponseEntity<TrackDto> updateTrack(@PathVariable Long id, @RequestBody TrackDto dto) {
    // Проверка прав в сервисе
    return ResponseEntity.ok(trackService.updateTrack(id, dto));
}

// В сервисе:
public TrackDto updateTrack(Long id, TrackDto dto) {
    permissionService.checkPermission(id, getCurrentUserId(), Permission.EDIT);
    // ... логика обновления
}
```

**Применимость**: При работе с операциями, требующими проверки прав доступа.
```

## 8. Метрики качества

### 8.1 Целевые показатели

| Метрика | Цель |
|---------|------|
| Повторение одной ошибки | ≤ 1 раз |
| Правил на категорию | 5-20 (поддерживать актуальность) |
| Устаревшие правила | Удалять/актуализировать |

### 8.2 Ревизия правил

Раз в месяц:
1. Проверить актуальность правил
2. Удалить дубликаты
3. Объединить похожие
4. Актуализировать примеры
