# Семантическая разметка (обязательные правила для ИИ-агентов)

## Ключевой принцип

**Код — это не просто реализация. Каждый важный блок имеет: адрес (anchor), смысл (contract), наблюдаемое поведение (logs). Агенту разрешено менять реализацию, но запрещено разрушать зафиксированную семантику.**

---

## 1. Иерархия приоритетов

При конфликте приоритет следующий (от высшего к низшему):

1. **Контракт** — зафиксированный смысл блока
2. **Инварианты** — то, что не должно нарушаться никогда
3. **Постусловия** — гарантии результата
4. **Текущая реализация** — код, который можно переписывать

**Если реализация расходится с контрактом, агент обязан приводить реализацию в соответствие контракту, а не наоборот.**

---

## 2. Обязательный контрактный слой

### 2.1 Что требует контракта

Контракт обязателен для:
- Каждой функции с бизнес-логикой
- Каждого метода класса, влияющего на состояние
- Каждого обработчика событий
- Каждого пайплайн-шага / трансформации
- Каждого интеграционного узла (API, DB, внешние системы)
- Каждого критичного бизнес-метода

### 2.2 Правило размещения контракта

**Принцип: сначала смысл, потом имя, потом реализация.**

Контракт размещается ДО объявления функции/класса/блока.

Если язык не позволяет docstring буквально до объявления — использовать ближайший допустимый формат предшествующего блока: комментарий, аннотацию, декларативный блок.

### 2.3 Обязательные поля контракта

| Поле | Обязательно | Назначение |
|------|-------------|------------|
| `ANCHOR` | Да | Уникальный идентификатор блока |
| `PURPOSE` | Да | Одна фраза: зачем существует этот блок |
| `@PreConditions` | Да | Что должно быть истинно ДО выполнения |
| `@PostConditions` | Да | Что гарантируется ПОСЛЕ выполнения |
| `@Invariants` | Да | Что не должно нарушаться НИКОГДА |
| `@SideEffects` | Да | Побочные эффекты (или "нет побочных эффектов") |
| `@ForbiddenChanges` | Да | Что запрещено менять семантически |
| `@AllowedRefactorZone` | Опционально | Что можно безопасно переписывать |

---

## 3. Формат контракта (шаблон)

### 3.1 TypeScript / JavaScript

```typescript
// [START_ANCHOR_ID]
/*
 * ANCHOR: ANCHOR_ID
 * PURPOSE: Одна фраза — зачем существует эта функция.
 * 
 * @PreConditions:
 * - условие 1
 * - условие 2 (или "нет нетривиальных предусловий")
 * 
 * @PostConditions:
 * - гарантия 1 при успехе
 * - гарантия 2 при ошибке (если применимо)
 * 
 * @Invariants:
 * - инвариант 1 (то, что истинно до, во время и после)
 * - инвариант 2
 * 
 * @SideEffects:
 * - побочный эффект 1 (или "нет побочных эффектов")
 * 
 * @ForbiddenChanges:
 * - что запрещено менять без явного согласования
 * 
 * @AllowedRefactorZone:
 * - что можно безопасно переписывать
 */
async function example(): Promise<void> {
  logLine("module", "DEBUG", "functionName", "ANCHOR_ID", "ENTRY", { args: "..." });
  // реализация
  logLine("module", "DEBUG", "functionName", "ANCHOR_ID", "EXIT", { result: "..." });
}
// [END_ANCHOR_ID]
```

### 3.2 Python

```python
# [START_ANCHOR_ID]
"""
ANCHOR: ANCHOR_ID
PURPOSE: Одна фраза — зачем существует эта функция.

@PreConditions:
- условие 1
- условие 2

@PostConditions:
- гарантия 1

@Invariants:
- инвариант 1

@SideEffects:
- побочный эффект 1 (или "нет")

@ForbiddenChanges:
- что запрещено менять

@AllowedRefactorZone:
- что можно переписывать
"""
def example():
    pass
# [END_ANCHOR_ID]
```

## 4. Связка контракта с якорями

### 4.1 Правило единого якоря

Каждый критичный блок логики имеет **ANCHOR_ID** — уникальный идентификатор в `UPPER_SNAKE_CASE`.

**ANCHOR_ID должен совпадать буквально в четырёх местах:**

1. `// [START_ANCHOR_ID]` — маркер начала блока
2. `ANCHOR: ANCHOR_ID` — внутри контракта
3. `// [END_ANCHOR_ID]` — маркер конца блока
4. `anchor` в `logLine()` — для привязки логов

### 4.2 Семантический блок как единое целое

по якорю агент должен уметь найти:
- Код (по START/END)
- Контракт (по ANCHOR: внутри)
- Логи (по anchor в logLine)

**Anchor — это адрес блока. Контракт — это смысл блока. Логи — это наблюдаемое поведение.**

---

## 5. Запреты на разрушительный рефакторинг

### 5.1 Категорически запрещено

- **Удалять "странную" логику без проверки её смысловой роли** — за ней может стоять неизвестное бизнес-ограничение
- **Менять поведение под видом cleanup** — рефакторинг не меняет семантику
- **Заменять бизнес-ограничения на "более стандартные" паттерны** — неочевидные ограничения могут быть критичны
- **Выдумывать недостающий смысл** — если он не закреплён явно, его нет
- **Менять контракт под реализацию** — только наоборот

### 5.2 Требование к рефакторингу

При любом рефакторинге агент обязан:
1. Проверить, не нарушены ли инварианты
2. Проверить, обеспечены ли постусловия
3. Проверить, не появились ли новые побочные эффекты
4. Проверить, не нарушены ли запреты из `@ForbiddenChanges`
5. Убедиться, что поведение соответствует контракту

### 5.3 Разрешённая зона рефакторинга

Если в контракте указан `@AllowedRefactorZone` — агент может менять только это.
Если не указан — агент может менять реализацию при условии сохранения семантики.

## 6. Работа при неполном контексте

### 6.1 Правило отсутствия догадок

**Если контракт, якорь или смысл блока неочевидны, агент НЕ ДОЛЖЕН молча додумывать поведение как ему кажется "логичным".**

### 6.2 Обязательные действия при неопределённости

Агент обязан:
1. **Сохранить текущую семантику максимально бережно**
2. **Минимизировать изменения**
3. **Явно пометить место как требующее уточнения:**
   ```python
   # @SEMANTIC_AMBIGUITY: Требуется уточнение смысла блока
   # @PRESERVED_BEHAVIOR: Поведение сохранено как есть без изменений
   ```
4. **Не разрушать существующие ограничения**

---

## 7. Работа с legacy-кодом

### 7.1 Принцип

**Если код плохой, устаревший или запутанный — это НЕ повод ломать его смысл.**

### 7.2 Процесс работы с legacy

1. **Извлечь или реконструировать контракт текущего поведения**
2. Зафиксировать контракт явно (даже если поведение кажется "неправильным")
3. Только после этого предлагать переписывание реализации
4. Любое изменение смысла требует отдельного согласования

### 7.3 Шаблон для legacy-кода без контракта

```python
# [START_LEGACY_BLOCK_ID]
# @LEGACY_CONTRACT_RECONSTRUCTED:
# PURPOSE: (описать наблюдаемое поведение)
# @PreConditions: (описать выявленные предусловия)
# @PostConditions: (описать выявленные постусловия)
# @Ambiguity: (что осталось неясным)
# @OriginalBehavior: (ссылка на оригинальный код или коммит)
# ...
# [END_LEGACY_BLOCK_ID]
```

## 8. Валидация соответствия контракту

### 8.1 Обязательная проверка после изменений

После генерации или правки кода агент ОБЯЗАН проверить:

- [ ] Соблюдены ли предусловия?
- [ ] Обеспечены ли постусловия?
- [ ] Не нарушены ли инварианты?
- [ ] Не появились ли новые побочные эффекты (не указанные в контракте)?
- [ ] Не нарушены ли запреты из `@ForbiddenChanges`?
- [ ] Соответствует ли поведение зафиксированному смыслу?

### 8.2 validator-agent

Если в проекте есть validator-agent, код ОБЯЗАТЕЛЬНО прогоняется через него на соответствие контрактам и якорям.

### 8.3 Формат отклонения от контракта

```python
# @CONTRACT_VIOLATION_DETECTED:
# - постусловие X не обеспечено
# - инвариант Y нарушен
# @CORRECTIVE_ACTION_REQUIRED: (описать требуемое исправление)
```

---

## 9. Явно запрещённые "упрощения"

Не считай разметку выполненной, если:

- Только блок `/* ANCHOR: … PURPOSE: … */` **без** пары `[START_…]` / `[END_…]`
- Только `PURPOSE` и `PreConditions` **без** `Invariants`, `SideEffects`, `ForbiddenChanges`
- `END_ANCHOR` как текст комментария **вместо** пары `[END_ANCHOR_ID]`
- Контракт над классом, но **у методов** нет своих контрактов
- Разные ANCHOR_ID в START, ANCHOR: и END
- Разные ANCHOR_ID в контракте и в logLine

---

## 10. Чеклист перед сдачей кода

### 10.1 Обязательные проверки

- [ ] У каждой новой/изменённой функции есть `[START_…]` … `[END_…]`
- [ ] Контракт содержит ВСЕ обязательные поля
- [ ] ANCHOR_ID в START, ANCHOR:, END и logLine — одинаковый
- [ ] Инварианты не нарушены
- [ ] Постусловия обеспечены
- [ ] Запреты из `@ForbiddenChanges` соблюдены
- [ ] Новые побочные эффекты отсутствуют (или отражены в контракте)
- [ ] Legacy-код обработан по правилу реконструкции контракта
- [ ] Неопределённость помечена `@SEMANTIC_AMBIGUITY` вместо догадок

### 10.2 Критерий готовности

Код считается готовым к ревью только при выполнении всех пунктов чеклиста.
