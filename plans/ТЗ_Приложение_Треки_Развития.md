# Техническое задание на разработку приложения «Треки Развития»

## 1. Общие сведения

### 1.1. Наименование системы
- **Полное наименование**: Веб-приложение «Треки Развития»
- **Краткое наименование**: TrackHub
- **Версия**: 1.0.0

### 1.2. Цель создания системы
Создать инструмент для управления личным и профессиональным развитием. Пользователи могут создавать Треки (направления развития), внутри которых ведётся работа с иерархическими заметками. Система должна поддерживать совместную работу через гибкое предоставление прав доступа.

### 1.3. Основные бизнес‑процессы
1. Регистрация и аутентификация пользователей
2. CRUD‑операции над треками развития
3. Управление древовидной структурой заметок внутри трека
4. Предоставление доступа к треку другим пользователям с разграничением прав (VIEW / EDIT)
5. Администрирование пользователей (только для роли ADMIN)

### 1.4. Роли пользователей

| Роль | Права |
|------|-------|
| **USER** | Управление своими треками и заметками. Предоставление прав на свои треки другим. Просмотр и редактирование чужих треков при наличии доступа (VIEW / EDIT). |
| **ADMIN** | Все права USER + просмотр списка всех пользователей, блокировка / удаление пользователей, **просмотр любого контента без явного VIEW‑доступа**. |

## 3. Функциональные требования (Backend)

### 3.1. Модуль аутентификации и регистрации

#### 3.1.1. Регистрация пользователя
- **Endpoint**: `POST /api/v1/auth/register`
- **Тело запроса**:
```json
{
  "username": "string (3‑50 символов, уникальный, только латиница, без пробелов)",
  "password": "string (минимум 8 символов, минимум 1 цифра, 1 заглавная буква, 1 специальный символ)"
}
```
- **Ответ**: `UserDto` (без пароля)
- **Права доступа**: Все пользователи
- **Валидация**: Bean Validation (`@Valid`) с ограничениями `@Size`, `@Pattern`, `@NotBlank`.

#### 3.1.2. Аутентификация пользователя
- **Endpoint**: `POST /api/v1/auth/login`
- **Тело запроса**:
```json
{
  "username": "string",
  "password": "string"
}
```
- **Ответ**:
```json
{
  "token": "JWT токен",
  "refreshToken": "токен обновления"
}
```
- **Права доступа**: Все пользователи
- **Валидация**: `@NotBlank` для обоих полей.

#### 3.1.3. Обновление access‑токена
- **Endpoint**: `POST /api/v1/auth/refresh`
- **Тело запроса**: `{ "refreshToken": "string" }`
- **Ответ**: новый `token` и `refreshToken`.
- **Права доступа**: Authenticated (по refresh‑токену).

#### 3.1.4. Выход из системы
- **Endpoint**: `POST /api/v1/auth/logout`
- **Права доступа**: Authenticated
- **Поведение**: отзывается `refreshToken`.

### 3.2. Управление пользователями (Admin)

#### 3.2.1. Получение списка пользователей
- **Endpoint**: `GET /api/v1/admin/users`
- **Параметры**: пагинация (`page`, `size`)
- **Ответ**: `Page<UserAdminDto>`
- **Права доступа**: ADMIN

#### 3.2.2. Блокировка/разблокировка пользователя
- **Endpoint**: `PATCH /api/v1/admin/users/{id}/lock`
- **Права доступа**: ADMIN

#### 3.2.3. Удаление пользователя
- **Endpoint**: `DELETE /api/v1/admin/users/{id}`
- **Действие**: Каскадное удаление треков и заметок владельца
- **Права доступа**: ADMIN

#### 3.2.4. Изменение роли пользователя
- **Endpoint**: `PATCH /api/v1/admin/users/{id}/role`
- **Тело запроса**: `{ "role": "ADMIN" | "USER" }`
- **Права доступа**: ADMIN

### 3.3. Управление треками

#### 3.3.1. Модель данных Track
```java
{
  "id": "Long",
  "title": "String (не пустое, max 200)",
  "description": "String (опционально)",
  "owner": "User (ManyToOne)",
  "createdAt": "Timestamp",
  "updatedAt": "Timestamp"

```

#### 3.3.2. Эндпоинты
| Метод | Путь | Описание | Права доступа |
|-------|------|----------|---------------|
| GET | `/api/v1/tracks` | Список треков текущего пользователя + треки с доступом. **Поддержка пагинации (`page`, `size`) и сортировки (`sort`)** | USER |
| POST | `/api/v1/tracks` | Создание нового трека. **Валидация `title` (не пустой, max 200)** | USER |
| GET | `/api/v1/tracks/{id}` | Получение деталей трека | VIEW / EDIT / OWNER |

|-------|------|----------|---------------|
| GET | `/api/v1/tracks/{trackId}/notes/tree` | Полное дерево заметок (DTO с children) | VIEW / EDIT |
| POST | `/api/v1/tracks/{trackId}/notes` | Создание заметки (корневой или дочерней). **Валидация `title`** | EDIT / OWNER |
| PUT | `/api/v1/notes/{id}` | Обновление заметки (валидация `title`) | EDIT / OWNER |

### 3.5. Совместный доступ (Sharing)

#### 3.5.1. Модель данных TrackPermission
```java
{
  "id": "Long",
  "track": "Track (ManyToOne)",
  "user": "User (ManyToOne)",
  "permissionType": "Enum (VIEW, EDIT)",
  "grantedAt": "Timestamp"
}
```
**Уникальный constraint**: `(track_id, user_id)`

#### 3.5.2. Эндпоинты
| Метод | Путь | Описание | Права доступа |
|-------|------|----------|---------------|
| GET | `/api/v1/tracks/{id}/permissions` | Список пользователей с доступом | OWNER |
| POST | `/api/v1/tracks/{id}/permissions` | Выдать доступ (username, permissionType) | OWNER |
| PUT | `/api/v1/tracks/{id}/permissions/{userId}` | Изменить тип доступа | OWNER |
| DELETE | `/api/v1/tracks/{id}/permissions/{userId}` | Отозвать доступ | OWNER |

### 3.6. Безопасность и ограничения
1. Все запросы, кроме регистрации/входа/обновления токена, требуют валидного JWT.
2. Проверка прав через `@PreAuthorize` или кастомный сервис авторизации:
   - **OWNER**: все права.
   - **EDIT**: обновление трека, CRUD заметок, но не управление правами и не удаление трека.
   - **VIEW**: только чтение трека и заметок.
3. **Bean Validation (`@Valid`)** применяется ко всем входным DTO.

## 5. Модель базы данных

### 5.1. SQL DDL (PostgreSQL)
```sql
-- Таблица пользователей
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(20) NOT NULL DEFAULT 'USER',
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Таблица треков
CREATE TABLE tracks (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    owner_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Таблица заметок
CREATE TABLE notes (
    id BIGSERIAL PRIMARY KEY,
    track_id BIGINT NOT NULL REFERENCES tracks(id) ON DELETE CASCADE,
    parent_id BIGINT REFERENCES notes(id) ON DELETE CASCADE,
    title VARCHAR(200) NOT NULL,
    content TEXT,
    order_index INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Таблица прав доступа
CREATE TABLE track_permissions (
    id BIGSERIAL PRIMARY KEY,
    track_id BIGINT NOT NULL REFERENCES tracks(id) ON DELETE CASCADE,
    user_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    permission_type VARCHAR(10) NOT NULL CHECK (permission_type IN ('VIEW', 'EDIT')),
    granted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(track_id, user_id)
);

-- Индексы для производительности
CREATE INDEX idx_notes_track_id ON notes(track_id);
CREATE INDEX idx_notes_parent_id ON notes(parent_id);
CREATE INDEX idx_permissions_user_id ON track_permissions(user_id);
CREATE INDEX idx_tracks_owner_id ON tracks(owner_id);
```
**Коды ошибок**: в `ErrorResponseDto` (см. раздел 6.2) описываются типовые коды (400, 401, 403, 404, 409, 500) и их форматы.
